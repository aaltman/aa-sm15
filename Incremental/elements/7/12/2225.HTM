Why Replication in First place ?

1) Availability - Allow system to work in case of partial failure

2) Reduce Latency - Keep Replicated data close to users location (Region based spread-out of data )

3) Scale out

Popular Mechanism to support Replication

1) Single-Leader Replication

2) Multi-Leader Replication

3) Leaderless Replication

Single -Leader Replication :

Most commonly used replication mechanism with Single Leader and multiple follower i.e. active/passive or Master-slave replica.

There are fairly large number of tools that supports Single-Leader Replication - MYSQL, Postgres, SQL etc..

Handling Node Outages with Leader in Single-Leader Replication:

Handling Node outages is challenging , especially when Leader Node is unavailable none of write can go into system, listed are few of steps to choose new leader in case of failover

a) Determining leader has failed - Choosing appropriate timeout for leader to respond

b)Choosing new leader based on election process - one simple strategy is to choose the replica with most up-to-date data changes.

c) Reconfigure the replicas with newly configured leader

When a new leader is elected there could be possible challenges /side-effects as listed below

a) Durability of Data - Potential Data Loss , This might be negligible and reason could be with synchronous/asynchronous mechanism used for replication

b)Split Brain - Previously chosen leader does not knows it is no more leader and system ends up with two leaders.

Even though automatic process exist to recover from leader failover , few prefer to do it manually considering above challenges.

Multi-Leader Replication

The mechanism allows more than one leader i.e master-master /active-active replication.

It allows more than one write into the systems from multi-data-center

Multi-Leader Replication is useful in clients with Offline Operation i.e you could think about Calendar/Meeting applications in Mobile Phone, All of the operations are stored in Mobile off-line and once device is connected to Internet the real sync would happen between different parties.

One other frequently usage is Collaborative Editing, for instance Google Docs, Etherpad allows multiple people to edit document.

Challenges with Multi-Leader Replication :

Resolving Conflict in Multi-Leader Replication is challenging, since system allows write to happen in more than one system , same document would potentially be edited by more than one party. To resolve conflicts there are few strategies listed below

a) Last Write Wins - Associate Timestamp with each write and pick the most recent one

b) Higher Unique ID Wins - Give each replica a unique ID and the one with higher version wins

c) Maintain conflict and let user to resolve .

CouchDB, MongoDB and other No-SQL DB are familiar systems when it comes to Multi-Leader Replication.

Leaderless Replication

It allows write to any of the replica and replica is responsible for broadcasting it to other nodes. There are no leaders and is often referred as Dynamo-Style mechanism.

Amazon Dynamo-DB, Cassandra, Riak are some example of leaderless replication.

Leaderless replication relies on quorum to ensure consensus.

When write happens to any of the replica system to ensure consistency of data Leaderless replication uses two approaches often

a) Read Repair- Read from several replicas and one with higher version is current data and ensure that is replicated to all other nodes

b) Anti-entropy process - Background process that constantly looks for differences in replica ad copies missing data to replica.

Summary :

Intention of post is to understand different replica mechanism that exist and use it wisely based on scenarios. There are well-established tools available in market to achieve listed replication mechanism and it all happens under-the-hood ,but its wise to know the difference that can help pick the best based on scenario/use-case
