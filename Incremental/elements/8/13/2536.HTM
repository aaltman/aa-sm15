: 3) How do we ensure reliable match creation?

In a system that prioritizes availability over consistency, we inherit an "eventual consistency" constraint when it comes to storing data such as swipes. If a "yes" swipe is stored, it might not be replicated to all nodes in a NoSQL system like Cassandra when a counterpart "yes" swipe is written, leading to a potential missed match.

To combat the entropy created from eventual consistency, we might consider caching "yes" swipes for a relatively short TTL to cover for the period of time when a DB like Cassandra is replicating data and / or undergoing anti-entropy. We only need to store "yes" swipes because these are the swipes that would actually trigger a match.

We'd store the swipes as just a list of user_id's in a distributed cache, partitioned by the swiper's user_id.
